## 4.1 Vue 响应式基础

::: info 什么是"响应式"？
简单说，就是**数据变了，页面自动更新**。

你改了一个变量的值，用到这个变量的地方会自动刷新显示。不需要你手动去更新 DOM。
:::

### 4.1.1 ref：让数据"活"起来

### 4.1.2 理解 ref

```javascript
// 创建响应式变量
const count = ref(0) 

// 在模板中直接使用 
{{ count }}  // 显示 0 

// 在 script 中需要用 .value
count.value++  // 变成 1
```

::: tip 为什么模板里不用 .value？
Vue 帮你自动处理了。在 `<template>` 里可以直接用 `count`，在 `<script>` 里要用 `count.value`。

这是 Vue 3 的设计，我们只需要关注数据，而Vue 会自动处理 DOM 更新。
:::

## 4.3 useFetch 详解

刚才我们用了 `useFetch`，它是 Nuxt 提供的数据获取工具，非常强大。

### 4.3.1 返回值说明

```javascript
const { data, pending, error, refresh } = await useFetch('/api/xxx')
```

| 返回值 | 类型 | 说明 |
|-------|------|------|
| `data` | `Ref` | 响应数据 |
| `pending` | `Ref<boolean>` | 是否正在加载 |
| `error` | `Ref` | 错误信息（如果有） |
| `refresh` | `Function` | 重新请求的函数 |

### 4.3.2 为什么用 await？

```javascript
// 这样写会等待第一次请求完成
const { data } = await useFetch('/api/xxx')

// 页面渲染时，data 已经有值了
```

::: tip 服务端渲染 (SSR)
Nuxt 默认开启 SSR，意味着页面在服务器上渲染好再发给浏览器。

`await useFetch` 会在服务器上等待数据加载完成，这样用户看到的页面直接就有数据，体验更好，对 SEO 也更友好。
:::

## 4.4 FastAPI 路由参数

现在后端只有一个固定的 API。如果我们想根据用户输入返回不同的数据呢？

### 4.4.1 路径参数

修改 `backend/src/backend/__init__.py`，添加新的 API：

```python
@app.get("/api/hello/{name}") # [!code focus]
async def hello(name: str): # [!code focus]
    return {"message": f"你好，{name}！欢迎来到全栈世界！"} # [!code focus]
```

现在访问 `http://localhost:8000/api/hello/张三`，会返回：

```json
{"message": "你好，张三！欢迎来到全栈世界！"}
```

::: info 路径参数语法
`{name}` 是路径参数的占位符，FastAPI 会自动把 URL 中对应位置的值赋给函数参数 `name`。

而且 FastAPI 会自动进行类型转换和验证！
:::

### 4.4.2 查询参数

再添加一个带查询参数的 API：

```python
@app.get("/api/greet") # [!code focus]
async def greet(name: str = "朋友", mood: str = "开心"): # [!code focus]
    moods = { # [!code focus]
        "开心": "😄", # [!code focus]
        "难过": "😢", # [!code focus]
        "生气": "😠", # [!code focus]
        "惊讶": "😲", # [!code focus]
    } # [!code focus]
    emoji = moods.get(mood, "😊") # [!code focus]
    return {"message": f"{emoji} 你好，{name}！今天{mood}吗？"} # [!code focus]
```

访问方式：
- `http://localhost:8000/api/greet` → 使用默认值
- `http://localhost:8000/api/greet?name=小明` → 指定 name
- `http://localhost:8000/api/greet?name=小明&mood=惊讶` → 指定 name 和 mood

::: tip 路径参数 vs 查询参数
- **路径参数**：`/api/hello/{name}` → `/api/hello/张三`
- **查询参数**：`/api/greet?name=张三` → 用 `?` 和 `&` 连接

一般来说，必须的参数用路径参数，可选的参数用查询参数。
:::

## 4.5 理解 $fetch

刚才我们用了 `$fetch`，它和 `useFetch` 有什么区别？

```javascript
// useFetch - 组件级别的数据获取，带响应式
const { data, pending, error, refresh } = await useFetch('/api/xxx')

// $fetch - 更底层的请求函数，适合在事件处理中使用
const response = await $fetch('/api/xxx')
```

::: tip 什么时候用哪个？
- **useFetch**：页面加载时获取数据，需要加载状态和错误处理
- **$fetch**：按钮点击等事件触发的请求，手动控制状态

简单说：自动的用 `useFetch`，手动的用 `$fetch`。
:::

### 4.6.1 什么是 Vibe Coding？

**Vibe Coding（氛围编程）** 是 AI 大神 Andrej Karpathy 在 2025 年初提出的概念。

简单说就是：**用自然语言告诉 AI 你想要什么，让 AI 帮你生成代码。**

```
传统编程：你写每一行代码
Vibe Coding：你描述需求，AI 写代码，你审核和调整
```

::: tip Andrej Karpathy 是谁？
- 前 Tesla AI 总监
- OpenAI 创始成员之一
- 斯坦福大学 CS231n（深度学习课程）讲师

他说的话在 AI 圈子里还是很有分量的。

### 4.6.3 Vibe Coding 的正确打开方式

::: warning ⚠️ 重要警告：AI 不是万能的！
我见过太多这样的情况：

- 让 AI 写了一大堆代码，自己完全看不懂
- AI 生成的代码有 bug，但自己不知道怎么改
- 过度依赖 AI，基础知识一点没学到
- 面试的时候被问基础问题，一问三不知

**AI 是工具，不是替代品。**
:::

正确的使用方式：

```
✅ 正确                          ❌ 错误
───────────────────────────────────────────
先理解原理，再用 AI 加速          直接让 AI 写，完全不看
让 AI 解释代码，帮助学习          复制粘贴，不管对错
用 AI 处理重复性工作              所有代码都让 AI 写
审核 AI 生成的每一行代码          盲目信任 AI 的输出
```

### 4.6.4 一个实际的例子

比如你想添加一个新功能："在页面上显示当前时间，每秒更新"

**传统方式**：自己查文档、写代码、调试...

**Vibe Coding 方式**：

```
你：帮我写一个 Vue 3 组件，显示当前时间，每秒自动更新。
    使用 Composition API 和 <script setup> 语法。

AI：好的，这是代码...（生成代码）

你：（审核代码）这里用 setInterval，组件销毁时需要清理吗？

AI：是的，需要在 onUnmounted 中清理...（解释并完善）
```

::: important 关键点
注意上面的对话：
1. **你知道要问什么**（Vue 3、Composition API、script setup）
2. **你能审核代码**（发现 setInterval 需要清理）
3. **你在学习**（通过和 AI 对话理解原理）

这才是 Vibe Coding 的正确姿势！
:::

### 4.6.5 给初学者的建议

::: note 建议
1. **前几章不要用 AI 写代码**——先把基础打牢
2. **遇到不懂的可以问 AI**——让它解释代码，帮助学习
3. **后面熟练了再用 AI 加速**——处理重复性工作
4. **永远保持好奇心**——AI 生成的代码，要知道它为什么这样写
:::

记住：**会用 AI 的程序员** 和 **只会让 AI 写代码的人**，是完全不同的两种人。

前者是驾驭工具的人，后者是被工具驾驭的人。

## 4.8 新手加餐：理解"页面动起来"背后的知识

这一章的重点不是按钮本身，而是理解现代前端应用的核心模式：**状态驱动界面**。

以前写静态 HTML 时，页面内容写死在文件里。现在使用 Vue 后，我们会先准备一些数据，再让页面根据数据自动变化。

```text
数据变了 -> Vue 发现变化 -> 页面自动更新
```

### 4.8.1 什么是状态？

状态就是页面当前记住的信息。例如：

| 状态 | 例子 |
| --- | --- |
| 计数状态 | 用户点了几次按钮 |
| 输入状态 | 输入框里现在写了什么 |
| 加载状态 | API 请求是否还在等待 |
| 错误状态 | 请求是否失败，失败原因是什么 |
| 数据状态 | 后端返回了什么内容 |

`ref` 就是用来创建状态的工具：

```js
const count = ref(0)
```

这句话可以翻译成：

```text
我创建了一个会被 Vue 追踪的变量，初始值是 0。
```

### 4.8.2 `v-if`、`v-else-if`、`v-else` 在解决什么问题？

请求后端 API 时，页面并不是瞬间拿到数据。它通常会经历三个阶段：

```text
正在请求 -> 请求失败 或 请求成功
```

所以页面也要对应三种显示：

```vue
<div v-if="pending">加载中...</div>
<div v-else-if="error">出错了：{{ error.message }}</div>
<div v-else>{{ data?.text }}</div>
```

这不是为了代码好看，而是为了让用户知道现在发生了什么。

::: tip 用户体验小原则
一个成熟的页面不应该只有"成功时长什么样"，还要考虑：

- 数据还没回来时显示什么？
- 网络失败时显示什么？
- 用户重复点击按钮时会不会出问题？
:::

### 4.8.3 `await useFetch` 和点击按钮里的 `$fetch` 有什么区别？

初学者很容易把它们混在一起。先记这个简单版本：

| 场景 | 推荐写法 |
| --- | --- |
| 页面一打开就要加载数据 | `await useFetch(...)` |
| 用户点击按钮后再请求 | `$fetch(...)` |

页面初始化数据：

```js
const { data, pending, error } = await useFetch('http://localhost:8000/api/greeting')
```

按钮点击后请求：

```js
async function refreshGreeting() {
  const result = await $fetch('http://localhost:8000/api/greeting')
  greeting.value = result.text
}
```

### 4.8.4 学会看报错，而不是害怕报错

::: warning 新手排错顺序
遇到页面不动，先不要乱改代码。按顺序检查：

1. 浏览器控制台有没有红色报错？
2. 前端终端有没有编译错误？
3. 后端终端有没有 Python 报错？
4. Network 面板里请求有没有发出去？
5. API 地址单独放到浏览器里能不能打开？
:::

### 4.8.5 小练习：给按钮加保护

当请求正在进行时，不希望用户疯狂点击按钮。可以给按钮加上：

```vue
<button :disabled="pending" @click="refresh">
  {{ pending ? '加载中...' : '刷新数据' }}
</button>
```

这背后的思想是：页面不仅要能用，还要防止用户误操作。
